# NPI build pipeline (v37 — per-detector Nugget Potential Index heatmaps)

Offline, reproducible build of the `data/npi/` heatmap tiles + tap-to-explain grid +
per-detector eval scorecard. All inputs are free / Australia-licensed; no API keys.

**v37** adds **stream micro-feature hotspots** (`stream_hotspots.py`) — the placer-science
sharpener. Real alluvial gold is 90% about *where on a creek*, not which creek, so v36's uniform
drainage term is post-processed into six placer signals and folded into the model as a `stream`
term. The VLF branch steals 0.18 of its weight from the uniform `drain` (drainage becomes the
*permissive* layer, the hotspot the *sharpener*); PI/ZVT take 0.05 (benches still matter). See
**"v37 stream hotspots"** below.

## Run (Mac/Linux with python3)

```sh
make deps      # one-time: numpy scipy pillow
make all       # fetch every input, then build   (or: make fetch && make build)
make rebuild   # drop the compute cache and recompute from the fetched inputs
```

Inputs and the compute cache are **gitignored** (large, regenerable). Only the pipeline
code and the built `../../data/npi/` output are committed. The cache (`cache_*.npz`) is
regenerated by the *same* functions that ship — there is **no orphan / cache-only path**
(this was the v33→v35 reproducibility fix). `make rebuild` proves it end-to-end.

## The v35 model — prior × evidence × interaction (not a linear distance sum)

The v33 heatmap was ~75–83% distance-to-goldfield-polygon, the three variants were
70–94% the same map, and the flagship magnetics signal wasn't even wired into PI. v35
rebuilds around a soft Bayesian-style prior modulated by real, decorrelating evidence:

```
evidence   = Σ detector-specific terms, each 0..1 (neutral 0.5 if missing)
interact   = detector-specific multiplicative confluence bonus
prior      = noisy-OR( goldfield-polygon prior , geological favourability litho×K )
prior_mod  = 0.15 + 0.85 · prior
burn_mult  = 1 + 0.10 · fresh_exposure
NPI        = 100 · GAIN · evidence · (1 + K_INT·interact) · prior_mod · burn_mult   (0..100)
```

### Real inputs

| Input | Source | Role |
|-------|--------|------|
| **Elevation** | AWS terrarium z12 (~30 m, SRTM-derived) | slope, convex-curvature (bedrock), D8 flow accumulation (drainage) |
| **Workings** | VicMine gold occurrences (WFS) | **2-D Gaussian KDE** (σ≈600 m local, σ≈2 km "hammered"), *not* a 500 m point-count |
| **Magnetics** | GA `magmap_v7_2019_RTP` + `_VRTP_AS` (WCS) | TMI-RTP = mineralisation (**+PI, −VLF**); analytic signal = structure/shears/contacts (**+PI, +ZVT**) |
| **Radiometrics** | GA `radmap_v4_2019_filtered_pctk` / `_ppmth` (WCS) | %K = granitic/K-alteration; Th = felsic host — rescues contact-hosted ground |
| **Geology** | Vic 1:250k `geol250_polygon` (WFS) | lithology favourability (granite/turbidite host vs basalt/cover) |
| **Fire** | Vic DELWP `fire_history` perimeters ≥2008 (WFS) | recency-weighted fresh-exposure multiplier |
| **Prior** | official goldfield + endowment polygons | soft distance prior, combined with geology |

Geophysics are **globally normalised** (p2–p98 across all regions), so the variants stay
comparable region-to-region. Missing data → **neutral 0.5**, never a silent bonus.

### The three detector variants (genuinely different maps — VLF/PI Pearson 0.85, was 0.94)

- **VLF (Gold Monster)** — gentle clean creek/eluvial ground: drainage-heavy, rewards **low** magnetics.
- **PI (GPX 6000)** — steeper mineralised hot ground: **high** magnetics + analytic-signal structure, drainage-blind.
- **ZVT (GPZ 7000)** — hammered premium ground: broad workings density + structure.

## v37 stream hotspots (`stream_hotspots.py`)

Six placer signals, computed on the **same flow route** as `drain` (one flood, cached), then
composited and upsampled. Bar-head (signal 6) is **deferred** — it needs sub-30 m LiDAR.

| Signal | w | What it flags |
|--------|---|---------------|
| **inside-bend / point-bar** | 0.30 | sagitta of the vectorised channel → the **inside** (deposition) bank of a high-curvature bend, where velocity is lowest and gold drops |
| **slope-break** | 0.25 | along-stream d(slope)/ds — steep-above / **flat-below** = the drop point where the current can no longer carry heavies |
| **bench / paleochannel** | 0.20 | cells ≤200 m from an active channel, **5–30 m above** local drainage, low local slope: old high terraces / deep leads |
| **confluence** | 0.15 | network nodes with in-degree ≥2, buffered ~50–220 m downstream with distance decay (the junction bar) |
| **pressure-shadow / pool** | 0.10 | the cell **immediately downstream of a >30 % slope drop** (relative), where a drop-pool forms and heavies settle |

Wired into the model as a `stream` term: **VLF 0.18** (stolen from `drain` 0.30→0.12) + a
hotspot-weighted confluence interaction, **PI 0.05**, **ZVT 0.05**. The result is that the
point-bar / slope-break on a creek out-scores the straight reaches — you can see *which meander to
sweep*, not just which stream (see the before/after eval).

### Resolution honesty

Hotspots run on the **~30 m SRTM** route (downsampled to a ~120 m flow grid — the same route
`drain` already uses, so the flood is paid once). Placer micro-features are usually **100 m+**
(meander wavelengths, terrace widths, confluence bars), so the sharpening is real at this scale.
**Bar-head scale (~10–30 m) is out of reach** and deferred — it would need an ELVIS 5 m DEM, which
has no open fetchable endpoint (documented in v35) and is infeasible full-region in pure Python.
The takeaway: v37 tells you *which bend / break / bench* on a creek, not *which end of one gravel bar*.

## Honest limitations (documented judgment calls)

- **DEM is SRTM-derived ~30 m, not 5 m LiDAR.** No Vic 5 m DEM is openly fetchable (ELVIS is
  interactive-order; GA endpoints 403) *and* full-region 5 m is infeasible here (multi-GB download
  + billion-cell pure-Python flow routing + the 60 MB tile budget). Terrain terms are coarse and
  weighted low; the audit's #1 problem was distance dominance, not terrain resolution.
- **Fire is a recency-weighted perimeter proxy, not per-pixel Sentinel-2 NBR severity** — COG
  band-math needs rasterio/gdal, absent in this build environment. Bushfires outweigh planned burns.
- NPI is a **heuristic prior, not a prediction.** Future versions retrain on confirmed gold events.

## Outputs (`../../data/npi/`)

| File | What |
|------|------|
| `{vlf,pi,zvt}/{z}/{x}/{y}.png` | palettised RdYlGn tiles, 3 detector variants, z10–12 native (~26 MB) |
| `npi-grid.png` | packed z8 (~490 m) grid, **7 stacked planes** for tap-to-explain: A=[npiVLF,npiPI,npiZVT]; B=[prior,workkde,drain]; C=[slopeDeg,bedrock,mag]; D=[asig,k,th]; E=[litho,burn,hammered]; **F=[stream,bend,slopebreak]; G=[confl,bench,pshadow]** (v37); **alpha = validity MASK only** (canvas premultiplies alpha — data must live in RGB) |
| `npi-meta.json` | bbox, grid geo-ref, weights, global-norm stats, limitations, per-spot component breakdown, eval |
| `npi-eval.json` | Settings scorecard: Pearson, per-input importance, 12 known spots + negative controls |
| `tiles-manifest.json` | every tile path, so the service worker precaches the region for offline use |

## Evaluation (`build_npi.py` prints + writes to npi-eval.json)

- **12 known trip spots** (want high) — Wedderburn 76 / Maldon 100 / Hepburn 97 / Avoca 90; the
  audit-flagged spots now light up for the right reasons: Mt Cole 12→24 (granite K + 2024 fire scar),
  Talbot 10→54 and Tarnagulla 38→42/45/58 (KDE workings), Chiltern 20→46, Whroo 18→39.
- **10 negative controls** (want <30) — basalt plains, urban Melbourne, Murray-Basin sediments.
  Worst = 25 (Ballan farmland); Melbourne CBD = 14. (Reservoirs flooding gold valleys — Cairn
  Curran, Eppalock — are excluded: they sit on real gold ground so a moderate score is correct.)
- **VLF↔PI Pearson** 0.846 (target <0.85) · **top input share** ≤30% per detector (target <40%).
